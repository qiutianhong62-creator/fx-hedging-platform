# 第四阶段：USD/CNY 历史市场数据与自动波动率设计

> 日期：2026-07-18
> 状态：用户已审阅并批准
> 项目：`fx-hedging-platform`

## 1. 阶段目标

第四阶段增加一个独立的 USD/CNY 市场数据模块。它自动读取 FRED 过去一年的每日汇率，计算一年期历史年化波动率，并返回最近一期公开市场参考汇率。

本阶段回答：

> 最近公开的 USD/CNY 参考汇率是多少？过去一年汇率实际波动有多大？这些数据来自哪里、截至哪一天、是否使用了缓存？

本阶段只提供历史市场数据摘要，不生成未来到期汇率预测，也不把最新参考汇率冒充未来预测。第五阶段负责机构预测，第六阶段再把机构预测和本阶段波动率接入已有概率模型。

## 2. 范围

### 2.1 本阶段支持

- 货币对固定为 `USD/CNY`。
- 公开数据源固定为 FRED 的 `DEXCHUS` 日度系列。
- 通过无需登录的 FRED CSV 地址自动读取过去 365 个自然日的数据。
- 清理 FRED 官方缺失值，验证日期与汇率。
- 使用每日对数收益率计算样本标准差。
- 按一年 252 个交易日换算年化历史波动率。
- 至少要求 200 个有效汇率观察值。
- 返回 FRED 最近一期有效汇率作为非实时市场参考值。
- 返回来源、观察窗口、样本数量、更新时间与缓存状态。
- 正常情况下 24 小时内复用本地缓存。
- FRED 暂时不可用时，最多使用抓取时间不超过 7 天的缓存。
- 最新市场观察值距离估值日不得超过 14 个自然日。
- 提供独立的只读 API，便于在 `/docs` 页面手工测试。

### 2.2 本阶段不支持

- 实时或盘中交易报价。
- 银行向特定客户提供的成交报价。
- 未来到期汇率预测或机构预测汇总。
- 隐含波动率或期权市场波动率曲面。
- 多币种、在岸与离岸人民币的并列分析。
- BIS 自动备用切换；代码边界保留未来增加数据源的能力。
- 数据库、后台定时任务或分布式缓存。
- 直接修改第三阶段概率接口的输入与输出。

## 3. 数据源选择

### 3.1 主数据源

使用 FRED `DEXCHUS`：Chinese Yuan Renminbi to U.S. Dollar Spot Exchange Rate。

- 来源机构：Board of Governors of the Federal Reserve System (US)。
- 发布项目：H.10 Foreign Exchange Rates。
- 单位：1 美元对应的人民币数量，与系统 `USD/CNY` 报价方向一致。
- 频率：每日观察值，H.10 通常每周汇总更新。
- 无需登录的 CSV：

```text
https://fred.stlouisfed.org/graph/fredgraph.csv
  ?id=DEXCHUS
  &cosd={开始日期}
  &coed={结束日期}
```

实际返回字段已经验证为：

```csv
observation_date,DEXCHUS
2026-06-01,6.7650
2026-06-02,6.7622
```

AnySearch 用于本阶段的数据源调查与官方资料核对。正式后端直接访问已经确认的 FRED CSV，不把 AnySearch MCP 作为网站运行时依赖，避免部署后依赖开发工具环境。

### 3.2 未选择的方案

- BIS：同样权威且有日度 CNY/USD 数据，但 SDMX 接口和字段结构更复杂，保留为未来备用来源。
- Frankfurter：无需登录且 JSON 简单，但属于中间聚合服务；第一版优先使用直接的官方系列。

参考：

- FRED 系列：https://fred.stlouisfed.org/series/DEXCHUS
- 美联储 H.10：https://www.federalreserve.gov/releases/h10/hist/
- BIS 备选：https://data.bis.org/topics/XRU/BIS,WS_XRU,1.0/D.CN.CNY.A

## 5. 组件边界

后端继续使用模块化单体结构，新增四个清晰单元：

```text
FRED客户端
  只负责构造URL、设置超时和下载CSV文本

历史数据解析器
  只负责把CSV转换成按日期排序的有效汇率

波动率计算器
  只负责从有效汇率计算每日收益率和年化波动率

市场数据服务
  负责编排缓存、抓取、验证新鲜度并生成响应
```

API 路由只调用市场数据服务，不包含下载、缓存或数学公式。

数据源访问定义为可替换接口。第一版实现 FRED 客户端，自动测试注入模拟客户端；以后增加 BIS 时不修改波动率计算器和公开 API。

## 6. 观察窗口与数据清理

### 6.1 请求日期

每次需要刷新时，以服务器估值日为结束日期，向前回溯 365 个自然日：

```text
开始日期 = 估值日 - 365天
结束日期 = 估值日
```

FRED 只会返回有观察值的日期，因此不要求每天都有一行。

### 6.2 清理规则

- 字段名必须包含 `observation_date` 和 `DEXCHUS`。
- 日期必须符合 `YYYY-MM-DD`。
- 汇率必须是有限且大于 0 的数字。
- FRED 官方空值、空字符串和 `.` 代表缺失观察，直接跳过。
- 周末和节假日没有观察值，直接保留自然缺口，不人为填充。
- 数据按日期升序排列。
- 同一日期重复时只保留最后一条有效记录。
- 格式错误、非数字或非正汇率不能静默忽略，整批数据视为上游格式异常。

### 6.3 数量与日期门槛

- 清理后至少有 200 个有效汇率观察值。
- 最新观察值距离估值日最多 14 个自然日。
- 观察值不足返回“历史有效数据不足”。
- 最新值超过 14 天返回“最新市场数据已经过期”。

## 7. 波动率公式

对按日期排序的有效汇率 `S`，相邻观察值生成每日对数收益率：

```text
rₜ = ln(Sₜ / Sₜ₋₁)
```

使用全部每日对数收益率的样本标准差，再换算为年化百分比：

```text
年化历史波动率(%) = 样本标准差(r) × √252 × 100
```

规则：

- 使用样本标准差，分母为 `n - 1`。
- 252 表示一年常用交易日数量。
- 不对周末和节假日生成零收益率。
- 纯计算器的结果必须是有限且大于等于 0 的数。
- 正式市场摘要要求波动率严格大于 0；恒定数据虽然数学上得到 0，但视为缺乏有效市场变化，不能送入第三阶段要求正波动率的概率模型。
- 返回值保留计算精度，未来前端负责显示位数。

FRED 页面可提供每日变化或连续复合变化，但不直接提供本项目定义的“一年滚动样本标准差年化波动率”，因此本项目根据公开公式自行计算。该结果是历史实现波动率，不是隐含波动率。

## 8. 缓存设计

### 8.1 缓存位置

第一版使用服务器本地 JSON 文件，不引入数据库。默认目录位于 Python 后端运行区域下的 `.cache/market-data/`，整个缓存目录加入 `.gitignore`，数据文件不提交 GitHub。

自动测试必须使用 pytest 提供的临时目录，不能读写开发者的真实缓存。

### 8.2 缓存内容

缓存保存：

- 数据源和系列编号。
- 查询开始与结束日期。
- 抓取完成的 UTC 时间。
- 已清理的日期与汇率观察值。

不把已经计算好的波动率当作唯一缓存内容，避免公式升级后继续使用旧计算结果。每次响应都从缓存观察值重新运行轻量计算。

### 8.3 缓存状态

响应使用三个稳定状态：

- `live_fetch`：本次成功访问 FRED 并更新缓存。
- `daily_cache`：缓存抓取时间不超过 24 小时，正常复用。
- `stale_fallback`：FRED 访问失败，使用超过 24 小时但不超过 7 天的缓存。

`stale_fallback` 必须同时返回 `is_stale: true`。前两个状态返回 `is_stale: false`。

### 8.4 安全写入

新缓存先写入同目录临时文件，完整写入后再原子替换正式文件，防止进程中断留下半个 JSON。损坏或字段不完整的缓存视为不存在。

## 9. 接口契约

### 9.1 请求

```text
GET /api/v1/market/usd-cny/history-summary
```

第一版不接收查询参数，观察窗口、数据源和公式均由后端固定，保证不同用户看到同口径结果。

### 9.2 成功响应

示例数值只说明结构，不作为固定测试结果：

```json
{
  "status": "available",
  "currency_pair": "USD/CNY",
  "quote_convention": "CNY per 1 USD",
  "market_reference": {
    "spot": 6.78,
    "observation_date": "2026-07-10",
    "is_live_quote": false
  },
  "historical_volatility": {
    "annualized_volatility_pct": 4.20,
    "method": "daily_log_returns_sample_stddev",
    "trading_days_per_year": 252,
    "window": "1y",
    "window_start": "2025-07-18",
    "window_end": "2026-07-18",
    "observation_count": 250,
    "return_count": 249
  },
  "source": {
    "provider": "FRED",
    "series_id": "DEXCHUS",
    "fetched_at_utc": "2026-07-18T02:00:00Z",
    "cache_status": "live_fetch",
    "cache_age_hours": 0,
    "data_age_days": 8,
    "is_stale": false
  }
}
```

接口必须明确表达：

- `market_reference` 是最近公开的每日参考值，不是实时成交价。
- `historical_volatility` 是过去一年计算结果，不是未来预测。
- `source` 让用户和开发者知道数据截至何时以及是否使用旧缓存。

## 10. 错误处理

统一错误形状：

```json
{
  "error": {
    "code": "market_data_unavailable",
    "message": "历史市场数据暂时不可用，请稍后重试"
  }
}
```

稳定错误类型：

- `market_data_unavailable`：FRED 网络失败、超时或非成功响应，且没有 7 天内缓存；HTTP 503。
- `market_data_invalid`：FRED CSV 字段或内容格式异常，或计算得到非正/非有限波动率；HTTP 502。
- `market_data_insufficient`：有效观察值少于 200；HTTP 503。
- `market_data_stale`：最新观察值距离估值日超过 14 天；HTTP 503。
- `market_cache_invalid` 不单独暴露给用户；损坏缓存先被忽略，再尝试 FRED，最终按上面的公开错误返回。

外部请求设置明确超时，不无限等待。第一版连接与读取总超时固定为 10 秒。

格式错误、数量不足或观察值过旧属于数据质量问题，不能用旧计算结果遮盖。只有纯网络或 FRED 服务暂时不可用时，才允许使用 7 天内缓存。

## 11. 测试与验收

### 11.1 CSV 解析测试

- 正常 CSV 解析出日期和正汇率。
- `.`、空值、周末自然缺口被正确处理。
- 数据按日期排序。
- 重复日期只保留最后一条有效记录。
- 缺少字段、无效日期、非数字、零和负汇率触发上游格式错误。

### 11.2 波动率计算测试

- 使用小型已知汇率序列手工核对每日对数收益率。
- 样本标准差与 `√252 × 100` 年化结果正确。
- 恒定汇率返回 0 波动率。
- 市场数据服务拒绝把 0 波动率生成正式摘要。
- 结果可重复，不使用随机数。
- 观察值少于 200 时拒绝生成正式摘要。

### 11.3 缓存与编排测试

- 无缓存时成功抓取 FRED、写入缓存并返回 `live_fetch`。
- 24 小时内重复请求不再次访问 FRED，返回 `daily_cache`。
- FRED 网络失败且缓存不超过 7 天时返回 `stale_fallback`。
- FRED 网络失败且缓存超过 7 天时返回 503。
- 损坏缓存被忽略并重新抓取。
- 新缓存使用原子替换，不暴露半写文件。
- 非网络类数据质量错误不能使用旧计算结果掩盖。
- 最新观察值超过 14 天时返回过期错误。

### 11.4 API 与回归测试

- `GET /api/v1/market/usd-cny/history-summary` 返回稳定成功结构。
- 四类公开错误返回稳定 HTTP 状态和中文信息。
- 响应明确包含 `is_live_quote: false`。
- 第一至第三阶段全部已有测试继续通过。

### 11.5 手工验收

自动测试使用固定模拟 CSV，不依赖外部网站。手工验收才真正连接 FRED：

1. 在本地启动 FastAPI。
2. 在 `/docs` 调用市场数据摘要接口。
3. 确认返回最近参考汇率、观察日期、样本数量和年化波动率。
4. 确认来源为 `FRED / DEXCHUS`，并标记不是实时报价。
5. 当天再次调用，确认缓存状态变为 `daily_cache`。
