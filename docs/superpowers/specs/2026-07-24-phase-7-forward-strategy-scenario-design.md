# 第七阶段：多笔远期策略固定汇率情景设计

> 日期：2026-07-24  
> 项目：`fx-hedging-platform`

## 1. 目标

新增一个独立的“远期策略固定汇率情景”接口。用户输入真实美元敞口、一个假设到期汇率，以及一笔或多笔银行远期报价，系统计算：

- 不套保时的人民币金额；
- 每笔远期带来的人民币金额变化；
- 整个远期策略的最终人民币金额；
- 相比不套保改善或恶化多少；
- 已套保、未套保和超额套保情况；
- 不套保与远期策略是否达到用户目标。

本阶段只验证确定性的远期现金流公式，不接入 ING 预测、FRED 波动率或概率模型。公式确认无误后，下一阶段再把同一个远期计算器接入自动概率分析。

## 2. 范围

### 2.1 本阶段支持

- 货币对仅支持 `USD/CNY`，解释为 1 美元对应多少人民币。
- 敞口类型支持美元应收、美元应付和持有美元。
- 用户可以输入一笔或多笔远期。
- 每笔远期只输入美元名义金额和银行提供的真实远期汇率。
- 所有远期自动使用敞口的统一到期日。
- 远期方向由敞口类型自动决定。
- 远期总金额可以小于、等于或超过真实敞口。
- 超额套保仍然计算，但必须返回超额金额和风险提示。
- 目标人民币金额保持可选。

### 2.2 本阶段不支持

- ING、FRED 或其他外部数据。
- 到期汇率概率、预计金额、概率区间和达标概率。
- 期权、掉期、货币市场套保或不同产品自由组合。
- 同一策略中使用不同到期日。
- 用户手工选择与真实敞口相反的远期方向。
- 远期提前平仓、展期、保证金、授信、交易费用或到期前市值。
- 市场参考远期价和银行报价优劣判断。

## 3. 为什么新增独立接口

保留现有不套保情景接口：

```text
POST /api/v1/analysis/no-hedge/scenario
```

新增远期策略接口：

```text
POST /api/v1/analysis/forward-strategy/scenario
```

不把远期字段塞进原接口，原因是：

- 原接口继续保持简单和兼容；
- 远期策略拥有自己的多笔交易、覆盖率和风险提示；
- 后续接入概率模型或期权时，不会让原接口不断膨胀；
- 两个接口可以使用相同数字分别测试，方便人工复算。

## 4. 用户输入

示例：

```json
{
  "currency_pair": "USD/CNY",
  "exposure_type": "usd_receivable",
  "notional_usd": 1000000,
  "maturity_date": "2026-11-15",
  "target_cny": 6680000,
  "assumed_maturity_spot": 6.60,
  "forward_legs": [
    {
      "notional_usd": 300000,
      "forward_rate": 6.80
    },
    {
      "notional_usd": 200000,
      "forward_rate": 6.75
    }
  ]
}
```

输入规则：

- `assumed_maturity_spot` 必须是有限且大于 0 的数。
- `forward_legs` 至少包含一笔远期。
- 每笔 `notional_usd` 和 `forward_rate` 必须是有限且大于 0 的数。
- 每笔远期不再重复填写币种、到期日和方向。
- `target_cny` 继续沿用现有规则：可以省略；填写时必须大于 0。
- 远期名义金额合计不设置为敞口金额的上限。
- 新接口中的敞口金额和目标金额同样必须是有限数；本阶段不借机修改原有接口的输入格式。

## 5. 远期方向

系统根据真实敞口自动匹配常见套保方向：

```text
美元应收    → 到期远期卖出美元、收取人民币
持有美元    → 到期远期卖出美元、收取人民币
美元应付    → 到期远期买入美元、支付人民币
```

第一版不允许用户逐笔选择买卖方向，避免把真实套保和额外投机方向混在一起。以后建立通用组合策略引擎时，再单独设计相反方向交易。

## 6. 核心公式

设：

```text
E = 真实美元敞口
S = 假设到期即期汇率
Ni = 第 i 笔远期美元金额
Fi = 第 i 笔远期汇率
```

不套保人民币金额：

```text
no_hedge_amount_cny = E × S
```

第 i 笔远期相对不套保的人民币金额变化：

```text
leg_difference_cny = Ni × (Fi - S)
```

整个策略的人民币金额：

```text
strategy_amount_cny
= no_hedge_amount_cny
+ Σ leg_difference_cny
```

这套公式同时适用于三类敞口，但经济解释不同：

- 美元应收和持有美元：结果表示人民币收入。正的差额代表收入增加，负的差额代表收入减少。
- 美元应付：结果表示人民币成本。正的差额代表成本增加，负的差额代表成本降低。

所有正式人民币金额使用现有金额计算规则，最终保留两位小数。中间计算不提前四舍五入。

## 7. 部分、完全和超额套保

定义：

```text
total_forward_notional_usd = Σ Ni
coverage_ratio = total_forward_notional_usd / E
unhedged_notional_usd = max(E - total_forward_notional_usd, 0)
overhedged_notional_usd = max(total_forward_notional_usd - E, 0)
```

`coverage_ratio` 在接口中使用小数，例如 `0.5` 代表 50%；前端以后负责显示为百分比。

覆盖状态：

```text
0 < 合计远期金额 < 敞口金额  → partial_hedge
合计远期金额 = 敞口金额      → full_hedge
合计远期金额 > 敞口金额      → over_hedged
```

由于接口至少要求一笔正金额远期，因此不会返回“零套保”状态；零套保继续使用原不套保接口。

超额套保的计算需要一个明确假设：

- 美元应收或持有美元时，超过真实敞口的远期卖出义务，假设在到期日按即期汇率买入缺少的美元后履约。
- 美元应付时，超过真实付款需求的远期买入美元，假设在到期日按即期汇率卖出多余美元。

因此超额部分仍使用 `Ni × (Fi - S)` 计算，但结果必须附带：

```text
warning_code = over_hedged
```

并明确说明实际交易可能带来额外资金、授信和方向性风险，本阶段未计算这些占用。

## 8. 结果设计

响应主要包含：

### 8.1 情景说明

- 情景类型为人工假设。
- `is_forecast = false`。
- 假设到期汇率。
- 自动决定的远期方向。

### 8.2 金额比较

- `result_kind`：人民币收入或人民币成本。
- 不套保人民币金额。
- 远期策略人民币金额。
- `strategy_minus_no_hedge_cny`：策略金额减去不套保金额的带符号差额。
- `economic_effect`：`improvement`、`worsening` 或 `no_change`。
- 对美元应付，成本下降才是 `improvement`；不能仅凭差额为正就判断有利。

### 8.3 每笔远期明细

每笔返回：

- 输入顺序编号；
- 自动方向；
- 美元名义金额；
- 银行远期汇率；
- 相对不套保的人民币金额变化；
- 对该敞口而言是改善、恶化还是无变化。

### 8.4 覆盖情况

- 远期美元金额合计；
- 覆盖比例；
- 未套保美元金额；
- 超额套保美元金额；
- 覆盖状态；
- 风险提示列表。

### 8.5 目标比较

如果用户填写 `target_cny`，分别返回：

- 不套保是否达标；
- 远期策略是否达标；
- 不套保距离目标的金额；
- 远期策略距离目标的金额。

不套保和远期策略分别复用现有 `TargetComparison` 结构。`difference_cny` 始终是非负的距离金额，具体属于收入盈余、收入缺口、成本节省或超额成本，由 `difference_type` 说明。

目标含义沿用现有规则：

```text
美元应收/持有美元 → 人民币收入达到或超过目标即达标
美元应付          → 人民币成本不超过目标即达标
```

如果没有填写目标，目标比较返回空值。

## 9. 人工复算案例

输入：

```text
美元应收：1,000,000 美元
假设到期汇率：6.60
远期 1：300,000 美元，远期汇率 6.80
远期 2：200,000 美元，远期汇率 6.75
目标金额：6,680,000 元
```

计算：

```text
不套保金额
= 1,000,000 × 6.60
= 6,600,000 元

远期 1 贡献
= 300,000 × (6.80 - 6.60)
= 60,000 元

远期 2 贡献
= 200,000 × (6.75 - 6.60)
= 30,000 元

策略金额
= 6,600,000 + 60,000 + 30,000
= 6,690,000 元
```

结果：

```text
相比不套保增加：90,000 元
覆盖比例：50%
未套保：500,000 美元
超额套保：0
不套保未达到 6,680,000 元目标
远期策略达到目标
```

相同数字用于美元应付时，策略成本为 6,690,000 元，相比不套保增加 90,000 元，因此经济效果是恶化而不是改善。

## 10. 后端结构

```text
远期策略请求
      │
      ▼
输入模型与检查
      │
      ▼
纯远期策略计算器
├─ 调用现有不套保金额逻辑
├─ 计算每笔远期差额
├─ 汇总策略金额
├─ 判断经济效果
├─ 统计覆盖与超额套保
└─ 比较目标金额
      │
      ▼
独立远期策略路由
```

计算器必须是纯函数：

- 不访问网络；
- 不读取缓存；
- 不依赖系统当前汇率；
- 相同输入始终返回相同结果；
- 可以在下一阶段被概率情景循环直接复用。

## 11. 错误处理

以下情况返回现有统一的友好输入错误：

- 假设到期汇率非正、无穷或不是有效数字；
- 没有任何远期交易；
- 任一远期美元金额非正、无穷或不是有效数字；
- 任一远期汇率非正、无穷或不是有效数字；
- 现有敞口、目标、币种或到期日输入不合法。

远期合计超过敞口不是输入错误，不阻止计算。

## 12. 测试

自动测试不访问任何外部网站，至少覆盖：

- 一笔远期的人工复算；
- 多笔远期的人工复算；
- 部分套保、完全套保和超额套保；
- 超额套保假设和风险提示；
- 美元应收、持有美元和美元应付；
- 对收入与成本正确判断改善或恶化；
- 远期汇率等于假设即期汇率时无变化；
- 目标金额存在和省略两种情况；
- 至少一笔远期及所有正数、有限数检查；
- API 成功结果和友好错误格式；
- 原有全部后端测试继续通过。

实现后在 `/docs` 页面使用第 9 节案例人工测试，并确认返回金额与手算完全一致。

## 13. 完成标准

用户可以输入一个真实 USD/CNY 敞口、一笔或多笔同日到期的真实银行远期报价和一个人工假设到期汇率。系统能够准确返回不套保金额、逐笔远期贡献、策略最终金额、经济效果、覆盖状态、超额套保风险和目标比较；结果能用简单乘法人工复算，且不依赖任何外部数据。
