# 第二阶段：不套保单一情景计算设计

> 日期：2026-07-18  
> 状态：已完成讨论，等待用户审阅书面规范  
> 项目：`fx-hedging-platform`

## 1. 阶段目标

第二阶段在第一阶段输入校验之上，增加一个单一假设到期汇率下的不套保金额计算。它回答：

> 如果到期时 USD/CNY 等于给定数值，在完全不套保的情况下，用户将获得多少人民币，或需要支付多少人民币？

这个汇率是后台计算和压力测试参数，不是系统预测，也不要求最终产品中的普通用户自行猜测。未来的市场预测模块可以生成多个汇率及其概率，并重复调用本阶段的同一个计算器。

## 2. 范围

### 2.1 本阶段支持

- 货币对固定为 `USD/CNY`，含义为 1 美元对应多少人民币。
- 敞口类型继续支持美元应收、美元应付和持有美元。
- 接收一个大于 0 的假设到期汇率。
- 计算单一情景下的不套保人民币金额。
- 根据敞口类型区分“预计获得人民币”与“预计支付人民币成本”。
- 用户填写目标金额时，判断该假设情景下是否达标，并解释差额。
- 用户没有填写目标金额时，只返回不套保金额，不生成目标比较。
- 金额内部使用十进制财务计算并按人民币分保留两位小数。

### 2.2 本阶段不支持

- 外部网站、机构预测或历史数据抓取。
- 系统对未来汇率的预测。
- 多个情景的批量接口和概率权重。
- 达标概率、期望值或概率分位数。
- 远期、期权或组合策略计算。
- 市场参考报价。

## 3. 用户可理解的计算逻辑

三种敞口均先计算：

```text
不套保人民币金额 = 美元金额 × 假设到期汇率
```

结果的业务含义不同：

```text
美元应收   → 预计获得人民币金额
持有美元   → 预计兑换获得人民币金额
美元应付   → 预计需要支付的人民币成本
```

示例：100 万美元、假设到期汇率 6.80，三种敞口的计算金额都是 680 万人民币，但应收和持有美元表示收入，美元应付表示成本。

## 4. 目标比较逻辑

目标比较只描述当前假设情景，不表示达标概率。

### 4.1 美元应收和持有美元

```text
不套保金额 ≥ 目标金额 → 达标
不套保金额 < 目标金额 → 未达标，差额是“还差多少”
```

金额高于目标时，差额解释为“超过目标多少”。

### 4.2 美元应付

用户填写的目标表示可接受的最高人民币成本：

```text
不套保成本 ≤ 目标最高成本 → 达标
不套保成本 > 目标最高成本 → 未达标，差额是“超出成本多少”
```

成本低于目标时，差额解释为“比目标节省多少”。

### 4.3 差额类型

后端使用稳定类型表达差额，前端再翻译成人类可读文字：

- `on_target`：恰好达到目标。
- `surplus`：应收或持有美元结果超过目标。
- `shortfall`：应收或持有美元结果低于目标。
- `cost_saving`：应付成本低于目标最高成本。
- `excess_cost`：应付成本超过目标最高成本。

## 5. 后端结构

本阶段继续使用模块化单体 FastAPI 后端，并将计算与网页接口分开：

```text
POST /api/v1/analysis/no-hedge/scenario
                    │
                    ▼
       请求模型与输入校验
                    │
                    ▼
       不套保单情景计算服务
                    │
                    ▼
       稳定的结构化响应模型
```

组件边界：

- 请求与响应模型：说明接口接收和返回哪些字段。
- 纯计算服务：只接收已校验数据并返回计算结果，不依赖 FastAPI、数据库或外部网站。
- API 路由：把请求交给计算服务并返回结果，不包含金额公式。
- 现有错误转换器：继续提供稳定的中文字段错误。

纯计算服务以后既可以被单一情景接口调用，也可以被概率模型循环调用，不需要复制金额公式。

## 6. 接口契约

### 6.1 请求

`POST /api/v1/analysis/no-hedge/scenario`

```json
{
  "currency_pair": "USD/CNY",
  "exposure_type": "usd_receivable",
  "notional_usd": 1000000,
  "maturity_date": "2030-12-31",
  "target_cny": 6800000,
  "assumed_maturity_spot": 6.75
}
```

请求沿用第一阶段规则，并增加：

- `assumed_maturity_spot` 必填且必须大于 0。
- 本阶段不设置武断的正常汇率上下限，以允许极端压力测试。

### 6.2 响应

```json
{
  "status": "calculated",
  "quote_convention": "CNY per 1 USD",
  "scenario": {
    "scenario_type": "assumption",
    "is_forecast": false,
    "assumed_maturity_spot": 6.75
  },
  "result_kind": "cny_proceeds",
  "no_hedge_amount_cny": 6750000.00,
  "target_comparison": {
    "target_cny": 6800000.00,
    "target_met": false,
    "difference_cny": 50000.00,
    "difference_type": "shortfall"
  }
}
```

`result_kind` 只取：

- `cny_proceeds`：美元应收或持有美元所得人民币。
- `cny_cost`：美元应付所需人民币成本。

`target_cny` 未填写时，`target_comparison` 为 `null`。

`scenario_type: assumption` 和 `is_forecast: false` 必须始终返回，以防未来将用户假设误解为市场预测。

## 7. 数字精度

- 内部金额和汇率计算使用 Python `Decimal`，不直接用二进制浮点数做正式乘法。
- 外部传入数字先通过其十进制文本含义转换，避免把已有浮点误差带入计算。
- 人民币金额使用四舍五入到小数点后两位。
- JSON 中金额以已经四舍五入到人民币分的数字返回；JSON 可能省略末尾的 `0`，前端负责按人民币格式展示逗号和固定两位小数。

## 8. 错误处理

除第一阶段已有错误外，本阶段新增：

```text
assumed_maturity_spot ≤ 0
→ 假设到期汇率必须大于 0
```

第一阶段已有规则继续生效：

- 货币对必须是 `USD/CNY`。
- 敞口类型必须是美元应收、美元应付或持有美元。
- 美元金额必须大于 0。
- 到期日必须晚于今天。
- 目标金额若填写则必须大于 0。

错误继续使用统一的 `validation_error` 响应格式。

## 9. 测试与验收

自动测试至少覆盖：

- 美元应收的不套保金额和收入含义。
- 持有美元的不套保金额和收入含义。
- 美元应付的不套保成本和成本含义。
- 应收或持有美元的目标恰好达标、超过目标和低于目标。
- 美元应付的目标恰好达标、节省成本和超出成本。
- 未填写目标时不生成目标比较。
- 假设汇率为 0 或负数时返回中文错误。
- 小数乘法按人民币分正确舍入。
- 第一阶段后端测试和已有套保计算测试继续通过。

手工验收通过 FastAPI `/docs` 页面完成：用户输入一笔敞口和单一假设汇率，能够看见不套保金额、结果含义以及可选的目标比较。

## 10. 后续连接方式

下一阶段可以在不修改本阶段金额公式的前提下扩展：

```text
机构预测 + 历史数据
          │
          ▼
多个到期汇率及其概率
          │
          ▼
重复调用本阶段纯计算服务
          │
          ▼
预计金额、达标概率和概率区间
```

因此，本阶段是未来概率分析的计算基础，但自身不包含任何预测结论。
